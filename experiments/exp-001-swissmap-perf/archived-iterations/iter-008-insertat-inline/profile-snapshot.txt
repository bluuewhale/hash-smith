Hot Path Analysis — insertAt() on tombstones==0 fast path:
  insertAt body:
    if (isDeleted(ctrlAt(ctrl, idx))) tombstones--;  // ALWAYS FALSE on fast path
    setEntryAt(idx, key, value);
    setCtrlAt(ctrl, idx, h2);
    size++;
    return null;

  Dead code on tombstones==0 fast path:
    - isDeleted() check → always false (tombstones==0 means no DELETED slots)
    - ctrlAt() call → memory read that is never acted upon
    - tombstones-- decrement → dead
    - Method call boundary itself → stack frame push/pop, register save/restore

  Expected gains from inlining:
    1. Eliminate method call overhead (saved JVM stack frame)
    2. Eliminate dead isDeleted branch + ctrlAt load
    3. Better register allocation: JIT can see full fast-path as single compilation unit
    4. Potential for better instruction scheduling across the entire fast path
